An artifact is one HTML file that must serve every size it's shown at: the
1×1 glance tile, a wide strip, a tall feed, and the full-screen view, with
nothing but its own inline CSS and the width and height it's given. This
page is the authoring guide; the normative rules live in the
[widget standard](/docs/reference/widget-standard).

## One file, no network

The artifact renders inside a sandboxed iframe with no network: no CDNs,
no web fonts, no `fetch`, no images by URL. Anything external simply
breaks. Inline everything. This is what makes an artifact durable: it
renders identically today and in the version browser a month from now.

## Design each size tier

Because the iframe _is_ the widget body, plain `@media` queries respond to
the widget's size. No postMessage, no resize observer. The standard
defines [the breakpoints](/docs/reference/widget-standard); the craft is
designing each tier rather than letting one be a cropped version of the
next one up:

- A **1×1** shows the essence: the one number that matters, centered.
- **2 columns** adds a KPI row and a terse ledger.
- **Wider** means columns of detail, not longer lines.
- The **full view** (≥ ~900 px) reads like a page, top-anchored.

Tiles never scroll, so content must fit the height at every tier: a list
that doesn't fit degrades to fewer items plus a visible `+N more` line.
Gate any fit-to-height measurement on `html[data-steward-tile]` so the raw
page and the full view keep every row; the full view is where `+N more`
actually lives.

## Theme tokens, injected

Author in the canonical gruvbox-dark palette as CSS custom properties
(`--color-*`), with `color-scheme: dark`, and paint via the tokens only.
When a viewer picks another theme, the dashboard overrides those same
custom properties inside the iframe at render time; a hard-coded hex won't
retheme. The same injection carries the brand mono: lead `--font-mono`
with `"Geist Mono Variable"` and load nothing yourself.

## The design language

Artifacts compose from a shared component set rather than inventing
per-routine visuals, so a board of widgets from different routines reads
as one product. The `widget-artifact` skill carries the full `design.md`;
the vocabulary:

- **Shell**: flex-column body; tiles top-align content (only the one-row
  glance tier centers); footer pinned; page generosity gated on
  `:root:not([data-steward-tile])`.
- **Ledger rows**: aligned key columns, hanging indents, optional
  trailing values. The workhorse for lists.
- **The stat**: the 1×1 tier's single KPI.
- **Pills, dots, meters, sparklines**: state and trend, mirroring the
  chrome's own vocabulary.
- **Designed empty states**: missing data is part of the artifact, not an
  error.

Type floors: body and data at **14px**, section labels at **12px**,
nothing smaller. Earn hierarchy with weight and color, not by shrinking.

**No motion.** An artifact is glanced at, not watched: it must look
settled the moment it paints. Never animate chart geometry or values on
load; bars growing and numbers counting up read as flicker in a tile. If
something genuinely must move (it almost never does), keep it ≤ 200ms and
respect `prefers-reduced-motion`.

## Links go out, in new tabs

Anything the artifact names that lives elsewhere (a PR, an issue, an
event) is an anchor to it, with `target="_blank" rel="noopener"` as
[the standard requires](/docs/reference/widget-standard). The tile is
triage; the source system is the follow-through.

## Person-relative content

An artifact is authored once and rendered for whoever can see the board,
so "you" is a render-time fact, never the routine runner. Name a known
owner in the third person ("Daniel's Daily Plan"); publish shared
artifacts viewer-neutral and re-group per viewer only via the injected
`window.__STEWARD_VIEWER__`. The exact rules are in the
[widget standard](/docs/reference/widget-standard#person-relative-content).

## The receipt doctrine

Every run ends in publish, even for routines whose real output is action
(PRs opened, issues filed). For those, the artifact is the **receipt**:
what was done, what changed, what needs you, each item linking out. When a
run truly has nothing to say, an honest "Nothing to report (checked
&lt;when&gt;)" is still freshness signal. Author receipts around deltas,
never a bare "ran OK".

## Testing an artifact

Dry-run the routine (`steward run <slug> --dry` from your data repo
checkout) and the artifact opens in your browser from a local file. Resize
the window across the breakpoints to walk the tiers. Nothing is pushed;
see [Dry runs](/docs/guides/hosts-and-scheduling#dry-runs). The board's
exact framing (sandbox, theme injection, tile stamp) can only be seen by a
real publish.
